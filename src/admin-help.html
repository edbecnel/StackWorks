Admin Help (operator-only)

This page is for server operators and developers.

Do not share the admin URL or admin token with players.

What the Admin page does

  • Shows the lobby for an online server.
  • Deletes a room via DELETE /api/admin/room/:roomId.

Where to open it

The admin UI is part of the client (the static website build), not the server.

  • Local dev: http://localhost:8080/admin.html
  • Deployed client: https://stackworks.games/admin.html

Server URL (admin setting)

The Server URL is the public URL of the online server (the thing that serves /api/...).

Two-domain example:

  • Client/site: https://stackworks.games
  • Server/API (Render): https://api.stackworks.games

In the Admin page, set Server URL to https://api.stackworks.games.

Admin token

The server-side admin delete endpoint is protected by a shared secret: LASCA_ADMIN_TOKEN.

  • The Admin page sends it as header x-lasca-admin-token.
  • If the server is missing LASCA_ADMIN_TOKEN, the endpoint returns 404 (hidden).
  • Wrong token returns 403.

The Admin page masks the token by default; use Show/Hide to toggle visibility.

Build flags (emitting admin pages)

By default, production builds do not emit admin.html. To include it (and this help page), build with: 

npx cross-env VITE_EMIT_ADMIN=1 vite build

Deploy notes: Vite base path

GitHub Pages repo hosting usually needs a base like /StackWorks/. A real domain (root hosting) usually needs /.

This repo supports overriding the production base at build time using VITE_BASE.

GitHub Pages (default)

npm run build

Custom domain at root (future)

npx cross-env VITE_BASE=/ vite build

Custom domain root + admin pages

npx cross-env VITE_BASE=/ VITE_EMIT_ADMIN=1 vite build

(1) Acquire the stackworks.games domain name

You need to own stackworks.games before you can point it at Cloudflare Pages and a Render API. You can buy the domain from any domain registrar (or use Cloudflare Registrar).

Checklist

  1. Pick a registrar (examples: Cloudflare Registrar, Namecheap, Google Domains’ successor, etc.).
  2. Search for stackworks.games and purchase it.
  3. Enable auto-renew.
  4. Enable WHOIS privacy (if available) so your personal info is not public.
  5. Decide where DNS will live:
    • Easiest: use Cloudflare DNS (you’ll change nameservers at your registrar to Cloudflare’s nameservers).
    • Alternative: keep DNS at your registrar (you’ll create CNAME/A records there instead).

If you’re using Cloudflare Pages, hosting DNS in Cloudflare is usually the smoothest setup, because custom domain + SSL + DNS records are all in one place.

(2) Register with Cloudflare Pages (client/site)

Cloudflare Pages is where you deploy the static site (the Vite build). This is your player-facing domain and also where the admin pages live if you choose to deploy them.

Sign up

  1. Create a Cloudflare account.
  2. (Recommended) Add your domain stackworks.games to Cloudflare so Cloudflare can manage DNS.
  3. If you add the domain to Cloudflare DNS: update nameservers at your registrar to the nameservers Cloudflare gives you.

You can create a Pages project before you finish nameserver changes, but custom domains won’t fully work until DNS is active.

Troubleshooting: if nslookup shows DNS request timed out, try turning off your VPN and re-trying. Some VPNs block or replace DNS.

(3) Register with Render (server/API)

Render is where you deploy the online server (Express + WebSockets). You’ll typically serve the API from api.stackworks.games.

Sign up

  1. Create a Render account.
  2. Connect your GitHub account (or whichever git provider you use).

(4) Set up Cloudflare Pages (website + project)

On Cloudflare’s website

  1. Go to Workers & Pages → Pages → Create a project.
  2. Select your GitHub repo for this project.
  3. Choose the Production branch (commonly main).
  4. Set the build settings (see below).
  5. Deploy.
  6. Add your custom domain:
    • In the Pages project, open Custom domains and add stackworks.games (and optionally www.stackworks.games).
    • Follow Cloudflare’s DNS prompts (if DNS is hosted in Cloudflare, it’s usually automatic).

In this project (build settings)

Cloudflare Pages needs to run Vite and upload the dist folder.

  • Framework preset: Vite
  • Output directory: dist
  • Environment variables (Production):
    • VITE_BASE=/ (because you’re hosting at https://stackworks.games, not under /StackWorks/)

Build command options

  • Player site only (no admin pages emitted): npm run build
  • Player site + admin pages emitted: npm run build:admin

If you deploy the admin pages publicly, keep https://stackworks.games/admin.html private. Treat the admin URL like a secret.

(5) Set up Render (website + project)

On Render’s website

  1. Click New → Web Service.
  2. Select your GitHub repo for this project.
  3. Configure the service:
    • Root directory: server
    • Build command: npm ci
    • Start command: npm start
  4. Add environment variables:
    • LASCA_ADMIN_TOKEN (required to enable admin delete)
  5. Deploy.
  6. Add custom domain:
    • In the service settings, add api.stackworks.games.
    • Render will show a DNS record (often a CNAME to a onrender.com hostname). Create that record in your DNS.

In this project (server)

  • The server is in server/ and starts with npm start.
  • Render provides PORT; the server reads process.env.PORT.
  • For realtime play, the server uses WebSockets at /api/ws and can fall back to SSE.

Token tip: generate a long random token and store it in a password manager. Use the exact same value for Render’s LASCA_ADMIN_TOKEN and the Admin page token.

Important: the server start script runs tsx src/index.ts. That means Render must install devDependencies for the server build (or you’ll need to adjust the server to run compiled JavaScript instead).

Admin setup after deploy

  1. Open https://stackworks.games/admin.html
  2. Set Server URL to https://api.stackworks.games
  3. Paste the admin token (same value as server’s LASCA_ADMIN_TOKEN)
  4. Refresh lobby, then use row actions (Delete / Copy)

The Server URL setting is stored in your browser storage. It is not a global server setting. Each admin device/browser needs to set it once.
